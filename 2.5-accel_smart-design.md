# 2. 设计理念

## 2.1 smart 是什么

**accel_smart 是一颗新弹药**，和 accel_cubic、accel_brutal 并列。用户通过 `./accel algo switch accel_smart` 热切换，或在 acc.conf 里设为默认算法。

smart 根据链路实时状态自动切换三种行为模式：

```
链路状态         算法行为                              tc-bpf 行为
──────────────────────────────────────────────────────────────────────
GOOD（干净）     brutal 激进发包                      不克隆（放行）
LOSSY（噪音）    reno 自然行为（不激进）               克隆每个包（多倍发包）
CONGEST（拥塞）  BDP 收敛 + drain 排空 + pacing 均匀   不克隆（放行）
```

## 2.2 为什么需要 smart

brutal 的 80% ack_rate 钳制适合"已知链路有丢包"的场景，但它**不区分丢包类型**——噪音性丢包和拥塞性丢包都一视同仁地按速率发。在真拥塞时继续激进会加剧拥塞。

smart 解决的问题：**在 GOOD 时像 brutal 一样激进，在 LOSSY 时用冗余包补偿而不激进填满链路，在 CONGEST 时主动退让。**

## 2.3 关键洞察

**LOSSY 时不能 brutal + 多倍发包**：

```
brutal 按 rate 发（已填满链路）
+ 多倍发包（每包发 2 份）
= 实际流量 2× rate
= 打爆链路 → 真拥塞 → 恶性循环
```

**正确做法**：LOSSY 时让 TCP 自然退让（reno 行为），多倍发包补偿丢包。总流量 ≈ 链路容量，冗余补丢包。

# 3. 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│  accel_smart                                                │
│                                                             │
│  ① struct_ops (accel_smart.bpf.c)                          │
│     每 ACK 触发:                                            │
│     - 更新 5 秒 pkt_info 窗口（复用 brutal 逻辑）            │
│     - EWMA 算丢包率                                         │
│     - classify → GOOD / LOSSY / CONGEST                    │
│     - apply:                                                │
│         GOOD    → brutal 速率钳制（按 rate_mbps 全速）       │
│         LOSSY   → reno 行为（cwnd 自然增长，不激进）         │
│         CONGEST → BDP 收敛 + drain 排空队列 + pacing 均匀   │
│     - 写 smart_link_state map（全局链路状态）                │
│     - 更新 smart_state_count map（可观测）                   │
│                                                             │
│            ↓ BPF map（smart_link_state）                    │
│                                                             │
│  ② tc-bpf egress (accel_smart_dup.bpf.c)                  │
│     每个出站包触发:                                          │
│     - 读 smart_link_state → 当前状态                        │
│     - LOSSY → 端口过滤 → bpf_clone_redirect → 冗余包       │
│     - 其他状态 → TC_ACT_OK 放行                             │
│                                                             │
│  ③ Rust 端                                                  │
│     - 启动时: load struct_ops + attach tc-bpf + 写配置 map  │
│     - 运行时: 无后台线程，无 tick，无协调                     │
│     - 停止时: Drop 自动 detach tc + unregister struct_ops   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**两个 BPF 程序之间通过 `smart_link_state` map 通信。** Rust 端在加载 dup skeleton 时用 `reuse_fd()` 共享 map fd，不需要 pinning。

## 4.3 状态判断（classify）

```c
enum { LINK_GOOD = 0, LINK_LOSSY = 1, LINK_CONGEST = 2 };

static __always_inline __u32 classify(struct smart_priv *p,
                                       struct smart_config *cfg,
                                       __u32 srtt_us, __u32 min_rtt_us)
{
    /* RTT 膨胀比: (srtt / min_rtt - 1) × 100 */
    __u32 rtt_ratio = 100;
    if (min_rtt_us > 0)
        rtt_ratio = srtt_us * 100 / min_rtt_us;

    /* 单个极强信号 → CONGEST（安全优先）*/
    if (p->loss_ewma_bp >= cfg->loss_congest_bp)
        return LINK_CONGEST;
    if (rtt_ratio >= 100 + cfg->rtt_congest_pct)
        return LINK_CONGEST;

    /* 有丢包但 RTT 稳 → LOSSY（噪音性丢包特征）*/
    if (p->loss_ewma_bp >= cfg->loss_lossy_bp &&
        rtt_ratio < 100 + cfg->rtt_congest_pct)
        return LINK_LOSSY;

    /* 干净：丢包低于 LOSSY 阈值的一半（滞后区）*/
    if (p->loss_ewma_bp < cfg->loss_lossy_bp / 2)
        return LINK_GOOD;

    /* 滞后区：维持当前状态，避免来回抖动 */
    return p->state;
}
```

**EWMA 更新（每 ACK）**：

```c
/* alpha = 1/8, 单次抖动只贡献 12.5% */
__u32 total = acked + losses;
__u32 sample_bp = 0;
if (total > 0)
    sample_bp = losses * 10000 / total;
p->loss_ewma_bp = p->loss_ewma_bp
                 - (p->loss_ewma_bp >> 3)
                 + (sample_bp >> 3);
```

**滞后区设计**：

```
      0%        0.5%       1%                  15%
  ────|──────────|──────────|────────────────────|────
      │  GOOD    │ 滞后区   │     LOSSY          │ CONGEST
      │  确定    │ 维持当前  │     确定           │ 确定
      │          │          │                    │
      loss_lossy_bp/2    loss_lossy_bp      loss_congest_bp
```

## 4.4 行为执行（apply）

```c
static void apply(struct sock *sk, struct smart_priv *p,
                   struct smart_config *cfg,
                   const struct rate_sample *rs)
{
    struct tcp_sock *tp = tcp_sk(sk);

    if (p->state == LINK_GOOD) {
        /* brutal 全速 — 复用 brutal_update_rate 的核心逻辑 */
        /* 按 cfg->rate 计算 cwnd 和 pacing_rate */
        brutal_set_rate(sk, p, cfg->rate);
        return;
    }

    if (p->state == LINK_LOSSY) {
        /* 简化 reno：cwnd 自然增长，不激进填充链路 */
        /* tc-bpf 层会做多倍发包补偿丢包 */
        if (rs->losses == 0 && rs->acked_sacked > 0) {
            /* 没丢包: cwnd 每 RTT 涨 1（加法增长）*/
            __u32 incr = rs->acked_sacked / tp->snd_cwnd;
            if (incr < 1) incr = 1;
            tp->snd_cwnd = min_t(__u32,
                tp->snd_cwnd + incr,
                tp->snd_cwnd_clamp);
        }
        /* 丢包时不动 cwnd, TCP 栈的 ssthresh 机制会处理 */
        /* 不设 pacing_rate, 让内核默认行为 */
        return;
    }

    /* LINK_CONGEST: 真拥塞 — BDP 收敛 + drain 排空 + pacing 均匀
     *
     * 借鉴 BBR 三个核心思想，但不实现完整 BBR 状态机：
     *
     * 1. BDP 估算：cwnd 收敛到 delivery_rate × min_rtt，而不是
     *    盲砍 70%。如果实际带宽只降了一点，cwnd 降幅也小。
     *
     * 2. drain 排空：刚进入 CONGEST 的前 2 个 RTT，pacing 降到
     *    估算带宽的 50%，快速排空路由器积攒的包，让 RTT 回到
     *    min_rtt。这加速了 classify 检测到链路恢复。
     *
     * 3. pacing 均匀：drain 之后按 90% 估算带宽均匀发包，避免
     *    突发加剧拥塞。
     *
     * 不实现 BBR 的 probe_bw 周期（classify 的 EWMA 已覆盖
     * "检测恢复"的功能）和 ProbeRTT（内核的 tcp_min_rtt 已维护）。
     */
    __u64 bw = 0;
    if (rs->interval_us > 0 && rs->delivered > 0)
        bw = (__u64)rs->delivered * tp->mss_cache * USEC_PER_SEC
             / rs->interval_us;

    __u32 min_rtt = tcp_min_rtt(tp);

    if (bw > 0 && min_rtt > 0) {
        /* BDP = bw × min_rtt */
        __u64 bdp = bw * min_rtt / USEC_PER_SEC;
        __u32 target_cwnd = (__u32)(bdp / tp->mss_cache);
        if (target_cwnd < 4) target_cwnd = 4;

        /* cwnd 收敛到 BDP，只降不升（拥塞期间不涨 cwnd） */
        tp->snd_cwnd = min_t(__u32, tp->snd_cwnd, target_cwnd);

        /* drain 阶段：进入 CONGEST 后的前 2 个 RTT
         * pacing 降到 50% 快速排空路由器队列
         * 之后恢复到 90% 均匀发包 */
        __u32 since = (bpf_jiffies64() & 0xFFFFFFFF) - p->last_change;
        __u32 two_rtt_jiffies = min_rtt * 2 * CONFIG_HZ / USEC_PER_SEC;
        if (two_rtt_jiffies < 2) two_rtt_jiffies = 2;

        __u64 pace;
        if (since <= two_rtt_jiffies)
            pace = bw / 2;         /* drain: 50% 排空队列 */
        else
            pace = bw * 9 / 10;    /* cruise: 90% 均匀发包 */

        __u64 max_pace = sk->sk_max_pacing_rate;
        sk->sk_pacing_rate = max_pace ? min_t(__u64, pace, max_pace) : pace;
    } else {
        /* 无法估算 delivery rate 时 fallback: 保守砍 cwnd */
        tp->snd_cwnd = max_t(__u32, tp->snd_cwnd * 7 / 10, 4);
    }
}
```

**CONGEST 行为的设计决策**：

| 考虑过的方案 | 结论 | 理由 |
|-------------|------|------|
| 纯 cwnd × 70% | ❌ 太简陋 | 不估算真实容量，恢复慢 |
| 切到系统 BBR | ❌ 不可行 | struct_ops 不能运行时切换 cong_ops |
| 在 BPF 里实现完整 BBR | ❌ 过度设计 | BBR 1199 行 + 69 符号依赖，2.2 阶段已放弃 |
| BBR probe_bw 周期 | ❌ 不需要 | classify 的 EWMA 已覆盖"检测恢复" |
| BBR ProbeRTT | ❌ 不需要 | 内核 tcp_min_rtt() 已维护 |
| **BDP 收敛 + drain + pacing** | ✅ 最终方案 | BBR 三个核心思想，~20 行实现，0 个新字段 |
```

## 4.5 cong_control 回调（smart_main）

```c
SEC("struct_ops")
__u32 BPF_PROG(smart_main, struct sock *sk, __u32 ack, int flag,
               const struct rate_sample *rs)
{
    struct tcp_sock *tp = tcp_sk(sk);
    struct smart_priv *p = inet_csk_ca(sk);

    if (rs->delivered < 0 || rs->interval_us <= 0)
        return 0;

    /* 读配置 */
    __u32 zero = 0;
    struct smart_config *cfg = bpf_map_lookup_elem(&smart_config_map, &zero);
    if (!cfg) return 0;

    /* 1. 更新 5 秒 pkt_info 窗口（和 brutal 完全相同的逻辑） */
    update_pkt_info_window(tp, p, rs);

    /* 2. 从窗口聚合 acked/losses, 算 EWMA */
    __u32 acked_total, losses_total;
    aggregate_window(p, tp, &acked_total, &losses_total);
    update_ewma(p, acked_total, losses_total);

    /* 3. classify */
    __u32 srtt_us = tp->srtt_us >> 3;
    __u32 min_rtt_us = tcp_min_rtt(tp);
    __u32 new_state = classify(p, cfg, srtt_us, min_rtt_us);

    /* 4. 最小驻留时间 200ms */
    __u32 now = bpf_jiffies64() & 0xFFFFFFFF;
    if (new_state != p->state) {
        __u32 elapsed = now - p->last_change;
        /* HZ=1000 时, 200ms = 200 jiffies */
        if (elapsed >= 200) {
            /* 更新 state_count map */
            update_state_count(p->state, new_state);
            p->state = new_state;
            p->last_change = now;

            /* 写全局 link_state, tc-bpf 读这个 */
            __u32 *link_state = bpf_map_lookup_elem(&smart_link_state, &zero);
            if (link_state)
                *link_state = new_state;
        }
    }

    /* 5. apply */
    apply(sk, p, cfg, rs);
    return 0;
}
```

void BPF_PROG(smart_init, struct sock *sk)
{
    struct smart_priv *p = inet_csk_ca(sk);
    __builtin_memset(p, 0, sizeof(*p));
    p->state = LINK_GOOD;  /* 新连接默认 GOOD */

    struct tcp_sock *tp = tcp_sk(sk);
    tp->snd_ssthresh = TCP_INFINITE_SSTHRESH;

    /* socket 计数 +1 */
    __u32 zero = 0;
    __u64 *cnt = bpf_map_lookup_elem(&smart_socket_count, &zero);
    if (cnt) __sync_fetch_and_add(cnt, 1);

    /* GOOD 计数 +1 */
    __u64 *good = bpf_map_lookup_elem(&smart_state_count, &zero);
    if (good) __sync_fetch_and_add(good, 1);
}

SEC("struct_ops")
void BPF_PROG(smart_release, struct sock *sk)
{
    struct smart_priv *p = inet_csk_ca(sk);
    __u32 zero = 0;

    /* socket 计数 -1 */
    __u64 *cnt = bpf_map_lookup_elem(&smart_socket_count, &zero);
    if (cnt && *cnt > 0) __sync_fetch_and_sub(cnt, 1);

    /* 当前状态计数 -1 */
    __u32 state_key = p->state;
    if (state_key <= 2) {
        __u64 *sc = bpf_map_lookup_elem(&smart_state_count, &state_key);
        if (sc && *sc > 0) __sync_fetch_and_sub(sc, 1);
    }
}

    __uint(type, BPF_MAP_TYPE_ARRAY);
    __uint(max_entries, 1);
    __type(key, __u32);
    __type(value, struct dup_config);
} smart_dup_config SEC(".maps");

SEC("tc/egress")
int smart_dup(struct __sk_buff *skb)
{
    __u32 zero = 0;

    /* 只在 LOSSY 时克隆 */
    __u32 *state = bpf_map_lookup_elem(&smart_link_state, &zero);
    if (!state || *state != LINK_LOSSY)
        return TC_ACT_OK;

    struct dup_config *cfg = bpf_map_lookup_elem(&smart_dup_config, &zero);
    if (!cfg || cfg->ifindex == 0)
        return TC_ACT_OK;

    /* 只处理 TCP（检查 L3 协议 + L4 协议号）*/
    __u16 proto = skb->protocol;
    __u32 ip_hdr_len;
    if (proto == bpf_htons(ETH_P_IP)) {
        /* IPv4: IHL 字段在第 0 字节低 4 位 */
        __u8 ihl_byte;
        if (bpf_skb_load_bytes(skb, 0, &ihl_byte, 1) < 0)
            return TC_ACT_OK;
        ip_hdr_len = (ihl_byte & 0x0F) * 4;
        __u8 ip_proto;
        if (bpf_skb_load_bytes(skb, 9, &ip_proto, 1) < 0)
            return TC_ACT_OK;
        if (ip_proto != IPPROTO_TCP)
            return TC_ACT_OK;
    } else if (proto == bpf_htons(ETH_P_IPV6)) {
        ip_hdr_len = 40;
        __u8 next_hdr;
        if (bpf_skb_load_bytes(skb, 6, &next_hdr, 1) < 0)
            return TC_ACT_OK;
        if (next_hdr != IPPROTO_TCP)
            return TC_ACT_OK;
    } else {
        return TC_ACT_OK;  /* 非 IP 包, 放行 */
    }

    /* 端口过滤 */
    if (cfg->port_min > 0) {
        __u16 dport_be;
        if (bpf_skb_load_bytes(skb, ip_hdr_len + 2, &dport_be, 2) < 0)
            return TC_ACT_OK;
        __u16 dport = bpf_ntohs(dport_be);
        if (dport < cfg->port_min || dport > cfg->port_max)
            return TC_ACT_OK;
    }

    /* 克隆并重发到同一接口 egress */
    bpf_clone_redirect(skb, cfg->ifindex, 0);

    return TC_ACT_OK;  /* 原包继续正常发送 */
}

# 9. 热插拔行为

```bash
# 从 brutal 切到 smart
./accel algo switch accel_smart
# → sysctl 切换, 新连接用 smart struct_ops
# → tc-bpf 已在 attach（启动时就挂了）
# → 新连接的 smart_init 设 state=GOOD → tc-bpf 不克隆 → 正常

# 从 smart 切回 brutal
./accel algo switch accel_brutal
# → sysctl 切换, 新连接用 brutal
# → tc-bpf 仍在 attach, 但 smart_link_state 不再更新
# → 残留值 = 最后一个 smart 连接写的状态
# → 最差情况: 残留 LOSSY → 多发一些冗余包
# → brutal 连接的冗余包是额外带宽, 但不会导致问题
# → 所有 smart 连接关闭后 link_state 不再更新, tc-bpf 开销 ≈ 0

# 热切换不需要改 socket.rs — 和现有 algo switch 完全相同
```

**关于残留状态**：当从 smart 切走后，`smart_link_state` 保持最后值。如果最后值是 LOSSY，tc-bpf 会继续克隆 brutal 连接的包。这不会导致错误（冗余包只是浪费带宽），但不理想。

**解决方案**：在 `smart_release` 回调里，当最后一个 smart socket 关闭时（socket_count 降到 0），把 `smart_link_state` 重置为 GOOD。

## 10.1 CPU 开销

| 路径 | 每次开销 | 触发频率 |
|------|---------|---------|
| struct_ops cong_control (GOOD) | ~30ns（brutal 逻辑） | 每 ACK |
| struct_ops cong_control (LOSSY) | ~20ns（reno 逻辑较轻） | 每 ACK |

对比 brutal 的 ~25ns/ACK，smart 在 GOOD 时多 ~5ns（EWMA + classify），在 CONGEST 时多 ~15ns（加 BDP 估算 + pacing 计算）。绝对值极小。

# 13. 风险与避坑

## 13.1 风险清单

| 风险 | 级别 | 应对 |
|------|------|------|
| verifier 撞坑 | 中 | 复用 brutal 已验证模式, 预留 1 天 |
| map reuse_fd 在 libbpf-rs 的 API | 低 | libbpf-rs 0.26 支持, 有文档 |
| tc clsact 和 VPS 现有 qdisc 冲突 | 低 | 启动时检查, 冲突则跳过 tc-bpf (降级到无多倍发包) |
| EWMA 启动期不准 | 低 | 新连接默认 GOOD, 前 50 个样本不切状态 |
| `tcp_min_rtt` 为 0 | 低 | classify 里判 0 跳过 |

## 13.2 避坑清单

### 坑 1: LOSSY 不能用 brutal

原因见 §2.3。LOSSY 时用 reno 行为 + 多倍发包。

### 坑 2: cong_control 注册后 cong_avoid 不会被调用

struct_ops 注册了 `cong_control` 回调后，内核不会调用 `cong_avoid`。所以 LOSSY 状态的 reno 行为必须在 `cong_control` 里手动实现（见 §4.4）。

### 坑 3: 状态残留

从 smart 切走后，`smart_link_state` 保持最后值。在 `smart_release` 里，当 `socket_count` 降到 0 时重置为 GOOD。

### 坑 4: tc clsact 残留

accel 崩溃后 tc clsact qdisc 可能残留。在 `load_smart()` 时先检查并清理。

### 坑 5: CONGEST 时 delivery rate 为 0

连接刚建立或长时间空闲后，`rs->delivered` 可能为 0，导致 BDP 无法估算。此时 fallback 到 `cwnd × 70%`（见 apply 代码的 else 分支）。不能因为算不出 BDP 就什么都不做。

### 坑 6: drain 阶段 pacing 过低导致超时

drain 用 50% pacing 持续 2 个 RTT。在极高 RTT 链路（>500ms）上，2 个 RTT = 1 秒，50% pacing 持续 1 秒可能触发应用层超时。但这和 BBR 的 drain 阶段一致，BBR 在生产环境已验证安全。如果真有问题，可以把 drain 比例从 50% 提到 70%。
